## 命名规范概述

### Google Python Style Guide 核心原则

1. **模块名**: 小写字母，下划线分隔 (`module_name.py`)
2. **包名**: 小写字母，不使用下划线 (`packagename`)
3. **类名**: CapWords 驼峰命名 (`ClassName`)
4. **函数名**: 小写字母，下划线分隔 (`function_name`)
5. **变量名**: 小写字母，下划线分隔 (`variable_name`)
6. **常量名**: 全大写字母，下划线分隔 (`CONSTANT_NAME`)
7. **私有成员**: 单下划线前缀 (`_private_method`)
8. **内部使用**: 双下划线前缀 (`__internal_use`)

### JavaScript/前端命名规范

1. **文件名**: kebab-case (`file-name.js`)
2. **类名**: PascalCase (`ClassName`)
3. **函数名**: camelCase (`functionName`)
4. **变量名**: camelCase (`variableName`)
5. **常量名**: UPPER_SNAKE_CASE (`CONSTANT_NAME`)
6. **私有成员**: 下划线前缀 (`_privateMethod`)

## 重构优先级

### 高优先级（影响可读性和维护性）

#### 后端

1. **路由文件命名**
   - `database_aggr.py` → `database_aggregations.py`
   - `instance_aggr.py` → `instance_aggregations.py`
   - 所有 `*_form_view.py` → `*_forms.py`

2. **函数命名**
   - 移除 `api_` 前缀（在 API 路由中冗余）
   - 修复语法错误（`get_instances_aggregations` → `get_instance_aggregations`）
   - 移除 `_optimized` 后缀（实现细节）

3. **类命名**
   - 简化过长的服务类名
   - 统一后缀使用（Service/Manager/Coordinator/Adapter）

#### 前端

1. **文件命名**
   - 所有下划线命名改为 kebab-case
   - `permission_policy_center.js` → `permission-policy-center.js`
   - `chart_renderer.js` → `chart-renderer.js`
   - 等等

2. **类命名**
   - 确保所有类使用 PascalCase

3. **函数命名**
   - 确保所有函数使用 camelCase
   - 统一事件处理函数命名（handle 前缀）

### 中优先级（改善一致性）

1. **变量命名**
   - 避免缩写（db_type → database_type）
   - 统一布尔变量前缀（is/has/can）

2. **常量命名**
   - 确保所有常量使用 UPPER_SNAKE_CASE

### 低优先级（可选优化）

1. **注释和文档**
   - 为公共 API 添加 docstring
   - 统一注释风格

2. **代码组织**
   - 按功能模块重新组织文件结构

### Python 命名速查

| 类型 | 规范 | 示例 |
|-----|------|------|
| 模块 | snake_case | `user_service.py` |
| 包 | lowercase | `services` |
| 类 | CapWords | `UserService` |
| 函数 | snake_case | `get_user()` |
| 变量 | snake_case | `user_name` |
| 常量 | UPPER_SNAKE_CASE | `MAX_SIZE` |
| 私有 | _leading_underscore | `_internal_method()` |

| 类 | PascalCase | `UserService` |

| 常量 | UPPER_SNAKE_CASE | `MAX_SIZE` |
| 私有 | _leadingUnderscore | `_internalMethod()` |
