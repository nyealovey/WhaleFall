# 项目级审计作业手册

本手册用于指导 TaifishingV4 仓库开展「大范围项目体检」与「专项审计」（安全、性能、依赖治理等）。目标是周期性盘点全局风险、评估技术债，并形成行动闭环。本指南与《PR 代码审查标准》共享 P0～P3 严重级别定义，审计发现可直接映射到对应等级。

## 适用场景

- 版本发布前的全仓回归检查（建议至少每季度一次）。
- 安全或性能事件后的专项复盘。
- 架构调整、技术栈升级、依赖大规模更新前后的健康评估。
- 团队轮岗、新成员接手模块时的入场盘点。

## 审计角色与分工

| 角色 | 职责 | 交付物 |
| ---- | ---- | ------ |
| 审计负责人（Tech Lead） | 制定计划、分配检查域、整合报告 | 审计计划、最终汇报、风险台账 |
| 审计执行人（领域 Owner） | 实施检查、收集证据、提出整改方案 | 域内检查清单、问题记录 |
| 支持团队（QA/DevOps/安全） | 提供工具、数据、验证能力 | 工具报告、压测结果、扫描日志 |

## 审计流程

1. **规划阶段**
   - 明确审计目标（安全/性能/合规等）与边界（服务、模块、时间窗口）。
   - 列出资产清单：服务、定时任务、第三方依赖、环境变量、CI/CD 流程。
   - 定义时间线、参与人、沟通频率，建立共享空间（如审计看板）。
2. **基线采集**
   - 收集现有监控、CI 报告、测试覆盖率、依赖列表、命名脚本输出、变更历史。
   - 对关键配置执行备份：`env.development`、`env.production`、`Makefile`、`migrations/` 等。
3. **分领域检查**（详见“检查域与指标”）
   - 采用清单 + 自动化脚本（`ruff`、`bandit`、`pytest --cov`、依赖安全扫描、性能压测脚本）。
   - 对每个发现标注 P0～P3 级别，记录证据（日志、截图、路径、命令输出）。
4. **汇总与评审**
   - 编制《审计问题列表》和《整改计划》，包含负责人、截止日期、验证方式。
   - 向产品/技术负责人汇报，确认是否需要立即进入 Incident 流程（针对 P0）。
5. **整改跟踪**
   - 将问题分发给模块 Owner，跟踪进度，必要时建立专项分支或任务单。
   - 对完成的项进行验证（复测脚本、基准测试、配置 diff）并更新台账状态。
6. **复盘与知识沉淀**
   - 统计指标：问题数量、级别分布、平均处理时长、重复出现的模式。
   - 将经验反馈到《PR 代码审查标准》与日常开发流程，形成闭环。

## 严重级别与响应策略

沿用 P0～P3 定义，并补充项目级响应措施：

- **P0（阻断级）**：立即升级为 Incident，组建跨团队响应小组；在 24h 内提供临时修复或回滚方案，同时向管理层汇报。
- **P1（高优先级）**：纳入当前迭代的专项任务，设置不超过 1 个工作周的截止日期；需要验证计划与回归测试报告。
- **P2（中优先级）**：列入技术债看板，设定不超过 1 个迭代（2～3 周）的处理窗口；必要时拆分为多个子任务。
- **P3（低优先级）**：收敛为优化建议或后续研究项，定期复查是否升级。

## 检查域与关键指标

| 域 | 覆盖范围 | 主要指标/问题示例 | 工具/数据源 |
| --- | -------- | ---------------- | ----------- |
| **架构与依赖** | 服务拓扑、模块边界、第三方 SDK、数据库/缓存配置 | 双驱动依赖（`psycopg[binary]` vs `psycopg2-binary`）、重复服务、循环依赖 | `pyproject.toml`、`uv.lock`、架构图、`scripts/refactor_naming.sh` 输出 |
| **安全与合规** | 鉴权链路、密钥管理、网络暴露面、审计日志 | 明文凭证、弱 TLS、缺少 CSRF/JWT 配置、权限绕过 | `bandit`、SAST/DAST 报告、`nginx/` 配置、vault/secret 审计 |
| **性能与容量** | API QPS、异步任务、数据库/缓存、队列、静态资源 | 吞吐下降、队列堆积、慢查询、定时任务超时 | APM 指标、`make dev start` 压测脚本、数据库慢查询日志 |
| **可观测性与运维** | 日志、指标、告警、追踪、CI/CD | 缺少追踪 ID、告警阈值过低/过高、部署脚本不一致 | `docs/`, `Makefile`, Grafana/Prometheus、CI pipeline 日志 |
| **规范与可维护性** | 命名、目录结构、注释、国际化、脚本 | `_api` 命名、硬编码颜色、docstring 英文缺中文解释 | `AGENTS.md`、`./scripts/refactor_naming.sh`、`ruff`、人工走查 |
| **数据与迁移** | `migrations/`, `sql/`, 数据回滚策略、基线数据 | 未执行的迁移、回滚脚本缺失、测试夹具过时 | Alembic 记录、`tests/conftest.py`、数据库快照 |

## 审计交付物模板

1. **审计计划**
   - 目标、范围、时间线、参与人、沟通节奏。
   - 工具与资源列表（脚本、压测环境、监控面板）。
2. **问题清单/风险台账**（建议表格保存）
   - 字段：ID、域、描述、级别（P0-P3）、证据、负责人、计划完成日期、状态、验证信息。
3. **整改计划与回归验证**
   - 对 P0/P1 列出立即动作、负责人、预计影响范围。
   - 对 P2/P3 划入技术债待办，标记复查日期。
4. **总结报告**
   - 指标：问题数量、严重级别占比、平均处理时间、复发问题列表。
   - 改进项：流程/工具/规范层面的优化建议（例如强化 `make quality`、新增监控面板）。

## 自动化与工具建议

- **静态检查**：`ruff`, `bandit`, `mypy`, `semgrep`（按需）。
- **命名与结构**：`./scripts/refactor_naming.sh`, `python -m compileall app`（捕获循环依赖）。
- **依赖与许可证**：`uv pip list --tree`, `pip-audit`, 许可证扫描器。
- **性能与容量**：`locust`, `wrk`, 自研压测脚本、`pytest -m slow`。
- **可观测性**：Grafana/Prometheus 面板、日志平台查询、APM trace。
- **配置一致性**：对比 `env.development` vs `env.production`、`Makefile` vs `Makefile.prod`。

## 沟通与治理建议

- 建立共享的审计看板或文档（如 Notion/Confluence），确保所有人查看同一版本。
- 每个域设定“联系人 + 备援”，避免知识孤岛。
- 对重复出现的问题（如命名违规、缺失 HTTPS 配置）发起根因分析（RCA），并将改进措施写入《PR 代码审查标准》或开发模板。
- 审计完成后召开 30 分钟复盘会，确认下一次审计时间与改进项 owner。

## 与日常开发流程的衔接

- 将项目审计中识别的技术债拆解为 issue，安排到迭代计划，确保每次迭代至少解决一定比例的 P2/P3。
- 审计指标（如命名脚本告警数量、`make quality` 失败率）应反馈给 PR reviewer，指导日常把关。
- 若审计暴露新的规范需求（例如强制 HTTPS 配置开关），需同步更新《PR 代码审查标准》与现有脚本。

> 建议：在 release 前 4 周启动项目级审计，release 前 1 周完成整改验证；平时可针对安全、性能、依赖等单一主题滚动开展“轻量审计”，以降低集中爆发的风险。
