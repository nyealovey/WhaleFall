# PR 代码审查标准

本指南覆盖 TaifishingV4 仓库在「单个变更 / PR」场景下的评审目标、流程、检查清单与 P0～P3 严重级别划分，帮助 reviewer 与作者保持统一的质量基线。若需要对整个项目开展阶段性体检，请参考《项目级审计作业手册》。

## 适用范围与前置要求

- 适用对象：所有提交至主线或长期分支的 PR、补丁包、Cherry-pick 变更。
- 前置检查：作者须在提交评审前完成 `make test`、`make quality`、`pytest -m unit`（必要时）、`./scripts/refactor_naming.sh --dry-run`，并在 PR 模板中附验证步骤与截图/日志。
- 评审产出：每个问题必须标注 P0～P3 级别与处理建议，评审完成后需要记录“通过/需修改”结论。

## 评审核心目标

1. **保证功能正确**：需求与边界输入得到覆盖，回归风险可控。
2. **守住安全与合规底线**：避免泄露敏感信息、破坏访问控制或引入高危依赖。
3. **维持性能与可观测性**：性能退化被量化并可观测，关键日志/指标齐备。
4. **贯彻项目规范**：命名、目录、日志策略、国际化与文档遵循《AGENTS.md》。
5. **强化可维护性**：代码清晰、有充分测试，复杂度与职责拆分合理。

## 严重级别定义

| 级别 | 说明 | 处理时限 | 责任人 | 示例 |
| ---- | ---- | -------- | ------ | ---- |
| **P0（阻断级）** | 违反安全/合规或导致主流程不可用，CI 必须失败。 | 立即修复，禁止合并。 | 作者与 reviewer 联合处理，必要时召集紧急同步。 | 明文传输敏感数据、SQL 注入、破坏数据迁移、生产配置缺失。 |
| **P1（高优先级）** | 引发严重退化或破坏关键非功能需求，但有短期回滚/降级手段。 | 1 个工作日内修复，合并前解决。 | 作者负责修复，reviewer 复核。 | 性能下降 >10%、资源泄漏、错误异常处理、缺失必要测试。 |
| **P2（中优先级）** | 影响可维护性/扩展性/体验，但不阻碍当前交付。 | 3 个工作日内修复，可随迭代交付。 | 作者负责，reviewer 跟进。 | 命名不符规范、缺失 docstring、日志级别不当、重复代码。 |
| **P3（低优先级）** | 轻微问题或改进建议，可纳入技术债排期。 | 依据排期处理，可延后释放。 | 负责人评估优先级。 | 样式微调、重构建议、文案优化。 |

> 在评审工具中使用标签或模版标注 P0～P3，便于统计与追踪，严重问题需同步记录于项目审计台账。

## 评审流程

1. **准备阶段**：作者完成自测、文档更新、截图录屏，确保 PR 描述包含需求背景、风险、验证方式。
2. **审查阶段**：reviewer 阅读 PR 描述、相关 issue，结合 `git diff --stat` 了解改动范围；必要时本地运行关键用例。
3. **问题反馈**：按严重级别输出问题，需包含文件路径与行号（如 `app/__init__.py:283`）、复现步骤、修复建议。
4. **修复与复核**：作者根据优先级处理问题，每次 push 需附说明；reviewer 复核关键改动并更新问题状态。
5. **合并前确认**：确保 CI 全绿、质量门禁通过、无未解决的 P0/P1；若需豁免，须由负责人书面确认并在项目审计文档登记。

## 检查清单（按领域）

### 功能正确性
- 需求覆盖：关键路径、边界输入、错误分支是否自测并附证据。
- 数据一致性：迁移脚本、事务、锁机制、幂等设计是否全面，回滚策略是否清晰。
- 国际化：界面/文案遵循语言策略，无硬编码文字。

### 安全与合规
- 鉴权授权：敏感接口继承正确蓝图/装饰器；权限检查顺序正确。
- 数据保护：禁止硬编码密钥，HTTP 明文会话必须通过环境开关控制；敏感日志已脱敏。
- 依赖安全：新增依赖的许可证、版本、供应链风险已评估；避免 `psycopg[binary]` 与 `psycopg2-binary` 并存。

### 性能与可观测性
- 算法与 IO：循环、SQL、网络调用复杂度可接受；必要时使用缓存/批处理。
- 资源管理：连接池、文件句柄、线程/协程生命周期无泄漏。
- 日志指标：关键分支输出结构化日志、追踪 ID；监控/告警是否需更新。

### 代码质量与规范
- 命名与结构：遵守 snake_case/CapWords/kebab-case；禁用 `_api` 命名、硬编码颜色。
- 注释与文档：新增 docstring/注释使用中文，解释设计原因、限制条件。
- 风格检查：通过 `black`、`isort`、`ruff`，docstring 与类型注解符合 Google 风格。

### 测试与验证
- 覆盖率：关键模块须提供单元/集成测试，标注 `@pytest.mark.unit`/`integration` 等。
- 可靠性：测试避免依赖真实外部服务，Mock/fixture 需清理状态。
- 手工验证：在 PR 描述中列出操作步骤、预期结果、截图或 curl 日志。

### 文档与发布
- README/docs 更新：新增命令、配置、端口、API 需同步文档。
- 迁移与回滚：数据库脚本、`sql/` 示例保持一致；描述回滚方案。
- 部署清单：环境变量、依赖服务（Redis/PostgreSQL）变更需通知 DevOps。

## 审查技巧

- 先读需求/issue，再看代码，建立上下文。
- 利用 `git blame`、`pytest -k`、`ruff`、`bandit` 等工具提高效率。
- 对复杂改动可请求 pair review 或同步走查，必要时记录会议结论。

## 与作者沟通建议

- 问题描述客观、具体，引用文件路径+行号。
- 提供修复方向或伪代码，必要时附参考链接、项目约定。
- 区分阻断问题与优化建议，避免优先级错乱。

## 与项目级审计的衔接

- P0/P1 问题若暴露出系统性缺陷（如命名脚本形同虚设、环境变量策略混乱），需在发现当日同步至项目级《项目级审计作业手册》中的风险台账。
- 每次版本发布前，建议将近期 PR 评审中积累的 P2/P3 技术债纳入下一次项目审计计划，形成“变更级 → 项目级”的闭环。 

> 下一步：若需大范围巡检、专项安全审计或性能压测，请结合本指南的严重级别定义，参考《项目级审计作业手册》制定更全面的审计计划。
