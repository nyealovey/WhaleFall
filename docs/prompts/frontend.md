你现在是“资深 UI/UX 审计师 + 前端工程负责人 + 无障碍(Accessibility)专家 + 产品体验研究员”的组合角色。请基于我给你的仓库与页面实现，对当前前端做一次“查漏补缺式”的 UI/UX 评审：重点找问题与改进机会，不要泛泛而谈。

输入材料（当前本地仓库）：
- 模板：app/templates/**（含 base.html、组件/宏、页面模板）
- 样式：app/static/css/**（尤其 app/static/css/variables.css、components/pages）
- 脚本：app/static/js/**（尤其 page-loader、core/http、common 组件、各页面入口）
- 典型用户任务（如：登录→列表筛选→查看详情→导出/同步→查看任务结果）

你的目标：
1) 识别“用户能感知到”的体验问题（可用性、信息架构、视觉一致性、反馈、错误、性能、可访问性）。
2) 每条问题都要给出“证据 + 影响 + 优先级 + 可落地修复方案 + 如何验证已修复”。
3) 额外输出一份“UI 规范/组件规范缺口清单”（我应该补写成文档/规范，避免同类问题反复出现）。

约束（必须遵守）：
- 当前仅支持桌面端：不要提出新增移动端 @media 适配；但要检查常见桌面宽度（1024/1366/1440/1920）下是否溢出、表格是否可用。
- 视觉色彩必须来源于 variables.css / 设计 token：指出任何手写颜色导致的不一致风险（作为 UX 一致性问题）。
- 若页面使用 filter_card：所有搜索/下拉控件必须用 `col-md-2 col-12` 栅格，禁止写死像素宽度；发现违反要明确标注证据。

工作方法（必须按步骤输出）：
A. 先建立“关键用户路径”：列出 Top 5 用户任务（按频率/价值排序），并把每个任务拆成 3~8 个页面/步骤。
B. 按“启发式评审 + 证据取样”找问题：对每条问题必须提供可定位证据（模板/JS/CSS 文件:行号，或具体组件/函数名）。
C. 进行“体验一致性扫描”：同类交互（按钮、表单校验、toast、loading、空状态、错误）在不同页面是否一致。
D. 给出“最小可执行修复路线图”（最多 8 条行动项，每条可在 0.5~2 天落地）。

评审维度（逐项过一遍，不得跳过）：
1) 信息架构与导航
- 菜单/面包屑/当前页高亮是否清晰？用户是否能回答“我在哪、我能去哪、我刚做了什么”？
- 列表→详情→返回：是否保持筛选条件/滚动位置？是否有“丢上下文”的问题？
- 术语一致性：同一概念在不同页面是否用不同名字（造成认知负担）？

2) 视觉层级与一致性（Design Consistency）
- 主次按钮、危险操作、链接样式是否一致？是否存在“看起来像按钮但不可点”的元素？
- 间距、对齐、表格密度、字号层级是否一致？是否出现拥挤/过稀导致效率下降？
- 色彩与状态：成功/警告/错误/禁用/悬停/聚焦是否统一且可识别（特别是对比度）？

3) 表单与数据录入（效率 + 防错）
- 必填/可选是否明确？默认值是否合理？是否能减少输入（预填/记忆/快捷选项）？
- 校验时机：提交后才报错 vs 输入中提示；错误信息是否“可行动”（告诉用户怎么改）？
- 风险操作（删除/导入/同步/批量）：是否有二次确认、影响范围提示、可撤销/回滚路径？

4) 反馈与状态（系统可理解性）
- Loading：请求中是否有明确反馈（按钮禁用/加载指示/骨架屏）？是否存在“用户以为没点上”的情况？
- Empty state：空列表/无结果是否给出下一步建议（重置筛选/创建入口）？
- 成功反馈：操作完成后用户如何确认（toast/状态标签/列表刷新/高亮变化）？
- 错误反馈：错误是否分级（可重试/权限不足/参数错误/系统故障）且文案一致？

5) 数据表格与筛选（管理后台核心）
- 筛选控件是否可发现、可复位、可组合？排序字段是否可理解？分页是否可控？
- 表格列：信息密度是否合理？关键列是否固定/可视？长文本/ID 是否可复制？
- 导出/同步：是否有进度、耗时预期、后台执行与结果查看入口？

6) 可访问性（A11y）
- 语义化：表单 label、aria 属性、按钮可读文本、图标按钮是否有可访问名称
- 对比度与可读性：小字号、灰色文本、禁用态是否仍可辨识

7) 性能与体验（感知性能）
- 是否“全站加载过多 JS/CSS”导致首屏慢？是否按页面加载脚本？
- 是否存在重复请求/无缓存/资源未版本化导致缓存失效？
- 前端渲染卡顿：大表格、频繁重绘、事件绑定过多

8) 文案与国际化（体验细节）
- 文案是否统一（提示语、按钮动词、错误信息）？是否避免“技术化错误”直接暴露给用户？
- 时间/时区/数字格式：是否一致？是否明确单位与范围？

9) 前后端契约漂移对 UX 的影响（重点找“兜底导致的混乱”）
- 检查前端是否大量使用 `||` / `??` / `or` 类兜底解析（例如 `message || error || fallback`、`data.csrf_token ?? data.data.csrf_token`）：
  - 这类兜底可能掩盖后端返回不一致，最终表现为“同类错误提示不一致/不可预测”。
- 对每个兜底点：说明它在 UX 上造成的风险，并提出“收敛契约/渐进移除兜底”的策略（含埋点统计兜底命中率）。

输出格式（严格遵守）：
I. “关键用户路径”(Top 5)：每条 = 任务 + 步骤列表
II. “缺口地图”：按上述 1~9 维度逐条给：缺口描述 + 我应该补写的 UI/UX 规范小节标题
III. “UI/UX 问题清单”（按 P0/P1/P2 分组；每条包含）

- 根因：设计/交互/实现/契约/一致性哪个环节导致
- 建议：短期止血（0.5~2 天）/中期重构（1~2 周）/长期规范化（可选）

附加输出：兜底/兼容/回退/适配（前端侧）
- 位置：文件:行号
- 类型：兼容/防御/回退/适配
- 描述：例如“错误文案解析使用 || 兜底多个字段，导致同类错误提示漂移”
- 建议：例如“统一错误 schema 后移除兜底，并加埋点统计命中率/告警”
